O que é o Ciclo de Vida do Desenvolvimento?

O SDLC (Software Development Life Cycle) é o conjunto estruturado de fases que organiza a criação de um sistema. Clique em cada camada para explorar o que acontece em cada nível.

Filosofia / Manifesto Base conceitual

O nível mais abstrato

Aqui vivem os valores e princípios que guiam como uma equipe pensa sobre desenvolvimento. O Manifesto Ágil (2001), por exemplo, não é uma metodologia — é uma declaração de valores: indivíduos sobre processos, software funcionando sobre documentação extensa.

Manifesto Ágil Lean Thinking Empirismo Scrum
Modelos de Processo Como o ciclo é organizado

Estrutura do ciclo de vida

Os modelos definem como as fases do SDLC se relacionam: sequencialmente (Cascata), em ciclos rápidos com usuário (RAD clássico) ou com princípios ágeis aplicados ao próprio RAD (RAD + Ágil). Ágil não é um modelo — é uma filosofia que orienta como o RAD evoluiu.

Cascata Linear · Sequencial
RAD Iterativo · Timebox
Ágil Incremental · Sprints
Frameworks & Metodologias Como o trabalho é organizado na prática

Scrum, XP, Kanban, DSDM…

Os frameworks operacionalizam os modelos com papéis, cerimônias e artefatos concretos. O Scrum define Product Owner, Sprint Planning e Retrospectiva. O Kanban visualiza o fluxo com cartões. O XP prescreve práticas de engenharia como TDD e pair programming.

Scrum XP Kanban DSDM SAFe LeSS
Práticas de Engenharia Técnicas do dia a dia

O que os desenvolvedores fazem

No nível mais concreto estão as técnicas e práticas técnicas: TDD (Test-Driven Development), CI/CD (integração e entrega contínua), refatoração, code review e sessões JAD. Essas práticas podem ser usadas com qualquer framework — elas são ortogonais ao modelo adotado.

TDD CI/CD Pair Programming Sessões JAD Refatoração Feature Flags
Ferramentas & Artefatos O que é produzido e utilizado

Artefatos e ferramentas de suporte

São os documentos, protótipos, sistemas e ferramentas que dão suporte ao processo: backlog no Jira, repositório no GitHub, documentação no Confluence, protótipos no Figma, testes automatizados no JUnit. Os artefatos produzidos variam conforme o modelo — no Cascata há muita documentação antecipada; no Ágil, há software funcionando a cada sprint.

Jira / Trello GitHub / GitLab Figma Confluence Product Backlog

Ponto-chave: Ágil não é uma metodologia — é uma filosofia. Scrum é um framework. Cascata e RAD são modelos de processo. Eles operam em camadas distintas da hierarquia acima.

Comparativo

Cascata × RAD × RAD + Ágil

Cascata e RAD como foram concebidos. O RAD + Ágil mostra como o RAD sobreviveu e evoluiu — sem deixar de ser RAD.

Critério Cascata RAD RAD + Ágil
Abordagem geral Linear e sequencial; cada fase conclui antes de iniciar a próxima Iterativa por componentes; usuário participa ativamente das iterações Iterativa e incremental; mesma lógica do RAD mas com sprints curtos de 1–3 semanas e entrega contínua
Gestão de requisitos Fixos e totalmente especificados antes do desenvolvimento Priorizados em workshops JRP; podem evoluir por versão Gerenciados em backlog priorizado; refinados a cada sprint — ainda parte do processo RAD
Envolvimento do usuário Concentrado na fase de levantamento; ausente durante o desenvolvimento Intenso nas sessões JAD/JRP e na validação de protótipos Contínuo; o usuário ou PO valida a cada sprint — versão moderna das sessões JAD
Mecanismo de validação Testes formais ao final do ciclo de desenvolvimento Protótipos evolutivos validados em sessões com usuários Protótipos + testes automatizados + feature flags — validação empírica e contínua
Ciclo de entrega Entrega única ao final do projeto (meses ou anos) Entregas por versão (V1, V2…) em timeboxes de 60–90 dias Entregas contínuas via CI/CD; sprints de 1–3 semanas substituem o timebox longo
Adaptação a mudanças Baixa; mudanças geram retrabalho custoso Moderada; mudanças entre versões são admitidas Alta; mudanças absorvidas no próximo sprint — o timebox é curto o suficiente para corrigir o rumo
Documentação Extensa e antecipada; é pré-requisito para avançar de fase Moderada; foco em protótipos funcionais e diagramas essenciais Mínima necessária; código funcionando, testes e ADRs substituem documentos formais
Estrutura da equipe Funcional e hierárquica; papéis bem definidos por fase Pequenas equipes multifuncionais; usuários integrados no processo Squads autônomos e multifuncionais (modelo Spotify); herdados diretamente do SWAT Team do RAD
Riscos principais Sistema entregue sem valor real; feedback tardio; obsolescência Escopo expansivo; dependência da disponibilidade do usuário Dívida técnica acumulada; desalinhamento de produto se o PO não estiver engajado
Contextos ideais Sistemas críticos com requisitos estáveis (defesa, aeroespacial) Sistemas corporativos com usuário disponível (ERP, sistemas internos) Produtos digitais modernos: fintechs, SaaS, apps — qualquer produto que precise evoluir com o mercado
Exemplos reais Sistema de controle de voo da NASA; projetos licitatórios governamentais Sistemas de gestão bancária (anos 1990–2000); ERPs legados Nubank, Spotify, GitHub, Notion, Linear — entregam a cada dia ou semana com squads autônomos
Legado histórico Base de toda engenharia de software; ainda exigido em contratos regulados Ponte entre Cascata e Ágil; introduziu JAD, prototipação e timeboxing É o RAD vivo: os princípios de Martin (1991) sobreviveram absorvendo os frameworks ágeis

Clássico

"Funciona na minha máquina"

O código roda perfeitamente no computador do dev, mas no servidor de produção simplesmente trava. Isso acontece porque o ambiente local tem configurações, versões e variáveis diferentes do servidor real. É a frase mais dita na história da profissão — especialmente para o QA (Quality Assurance), a pessoa responsável por testar o sistema e que invariavelmente encontra o bug que "não existia". O QA reporta o erro, o dev abre no próprio computador e diz: "aqui tá funcionando." Esse ciclo tem nome: works on my machine. Foi a principal motivação para criar ferramentas como Docker, que empacotam o ambiente junto com o código.

💡 Daí surgiu a resposta irônica: "Ship the machine." — manda o computador do dev junto com o sistema.
🔍

Dev: "Tá pronto, pode testar."

QA 5 minutos depois:
"Encontrei 12 bugs."

QA na área

QA — o detetive do time

QA (Quality Assurance) é a pessoa ou equipe responsável por garantir que o sistema funciona como deveria antes de chegar ao usuário. O trabalho deles é tentar quebrar o que o dev construiu — e eles são muito bons nisso. A relação dev-QA é de amor e ódio: o dev odeia receber o relatório de bugs, mas sem o QA os problemas chegariam direto ao usuário final. O QA não é o inimigo — é o que separa o time de uma catástrofe em produção.

🛡️ Times sem QA dedicado geralmente descobrem os bugs pela pior via possível: o usuário reclamando.
🔥

"Não é um bug."

É uma feature não documentada.

Autodefesa

"Não é bug, é feature"

Quando algo no sistema se comporta de forma errada mas o dev não quer admitir — ou não tem tempo de corrigir — a saída clássica é declarar que aquilo é, na verdade, uma funcionalidade. Um bug é um comportamento incorreto do software. Uma feature é uma funcionalidade intencional. A piada está na inversão: o que deveria ser consertado vira algo que "estava planejado assim". É o equivalente a queimar o jantar e chamar de "culinária molecular".

😅 A frase completa que circula nos times: "Não é bug, é feature não documentada."
🙈

"Não é bug se o usuário não encontrar."

— Dev às 23h antes do prazo

Filosofia dev

"Não é bug se ninguém achar"

A lógica (incorreta, mas muito praticada) de que um bug só importa se alguém tropeçar nele. Na teoria, todo bug deveria ser corrigido. Na prática, com prazo apertado e lista de prioridades enorme, bugs "ocultos" ficam escondidos em cantos do sistema que ninguém visita — até o dia em que um usuário clica exatamente no lugar certo e tudo desmorona. O dev que disse isso provavelmente sabe exatamente onde o bug está enterrado.

⚠️ O problema é que o usuário sempre encontra. Sempre. Geralmente na pior hora possível.

Time de Produto

O time de produto e o "é simples"

O time de produto (Product Manager / PO) é responsável por definir o que o sistema deve fazer e priorizar funcionalidades. A tensão clássica com o time de desenvolvimento acontece quando o produto pede algo que soa simples — mas tecnicamente é meses de trabalho. "Fazer igual ao Google" ignora que o Google tem milhares de engenheiros e décadas de desenvolvimento. Pedir isso pra sexta é como pedir uma reforma completa de apartamento pra amanhã.

🤝 No RAD e no Scrum, o PO participa ativamente do processo justamente para alinhar expectativas antes de virar pedido impossível.

Frontend vs. Backend

Frontend é tudo que o usuário vê e clica: botões, animações, layout, cores. Backend é o que roda nos bastidores: banco de dados, regras de negócio, autenticação. Cada lado acha que o trabalho do outro é mais fácil. Quem faz os dois se chama Fullstack — e costuma ser cobrado como dois profissionais pelo salário de um.

💡 Este próprio site usa as duas partes: HTML/CSS (front) e JS com lógica (back leve).

Matemática do dev

99 bugs no código

Corrigir um bug muitas vezes cria dois novos — porque o código é interligado. Mudar uma parte pode afetar comportamentos que pareciam não ter relação nenhuma. Por isso existem os testes automatizados: scripts que verificam se o resto do sistema ainda funciona após cada alteração. Sem eles, consertar é como apertar um balão: some de um lado, aparece do outro.

💡 O RAD usava prototipação iterativa exatamente para pegar bugs cedo, quando ainda são baratos de corrigir.
💣

"Já foi, pessoal."

Dev que fez deploy
na sexta às 17h58.

Regra sagrada

Nunca faça deploy na sexta

Deploy é o ato de publicar uma nova versão do sistema para os usuários. Se algo der errado numa sexta à tarde, a equipe ou trabalha o fim de semana inteiro para corrigir, ou deixa o sistema quebrado até segunda. Virou uma regra não escrita da profissão. Quem já ignorou essa regra nunca mais ignora.

📅 Muitas empresas têm freeze periods oficiais: janelas proibidas de deploy perto de feriados e fins de semana.
